# Tech layoffs vs. LLM coding-model releases

A free, live, auto-refreshing static site that plots cumulative tech layoffs by industry over time, with timeline markers at every major LLM coding-model release (Anthropic, OpenAI, Google).

- **Hosting:** GitHub Pages — free, no server needed.
- **Data refresh:** GitHub Actions cron, **Mondays + Thursdays at 09:00 UTC** (≈ 2× / week).
- **Stack:** plain HTML + Chart.js + a tiny Python ETL. No build step.

---

## How it fits together

```
.
├── index.html                ← the live site (Chart.js, no framework)
├── data/
│   ├── data.json             ← regenerated by CI; consumed by index.html
│   ├── markers.json          ← curated LLM release timeline (edit freely)
│   └── raw_layoffs.csv       ← seed CSV used as fallback when no source URL works
├── scripts/
│   ├── build_data.py         ← fetches CSV, normalises, aggregates → data.json
│   └── sources.txt           ← prioritised list of CSV mirror URLs
└── .github/workflows/refresh.yml   ← cron + push, refreshes data.json and deploys
```

## One-time setup (≈ 5 minutes)

1. **Create a GitHub repo.** Name doesn't matter — let's say `llm-layoffs-tracker`.
2. **Push these files** to its `main` branch.
3. **Enable Pages.** Repo → *Settings* → *Pages* → *Source: GitHub Actions*.
4. **Done.** The first push triggers the workflow; within ~2 minutes your site is live at
   `https://<your-username>.github.io/<repo-name>/`.

The cron from then on refreshes `data/data.json` every Monday and Thursday and redeploys.

> **Tip:** you can manually re-run the refresh any time from the *Actions* tab → *Refresh data & deploy* → *Run workflow*.

## Editing the model-release timeline

`data/markers.json` is the source of truth. Each entry is just:

```json
{ "date": "2025-09-29", "family": "anthropic", "label": "Claude Sonnet 4.5" }
```

Add, remove, or re-date entries; the chart picks them up on the next page load. `family` controls the marker color — defined under `"families"` in the same file.

## Swapping the layoffs data source

The ETL tries sources in this order:

1. The `LAYOFFS_CSV_URL` env var (set as a [repo Variable](https://docs.github.com/en/actions/learn-github-actions/variables) called `LAYOFFS_CSV_URL` to override).
2. URLs listed in `scripts/sources.txt`, top-down.
3. The committed `data/raw_layoffs.csv` seed (so the chart never goes blank).

Any source must be a CSV with at minimum a date column and a "total laid off" column — `build_data.py` recognises common column-name variants (see `COLUMN_ALIASES`). The default URL is a community mirror of layoffs.fyi data; if it goes stale, swap in another mirror or your own CSV.

## Running locally

```bash
python3 scripts/build_data.py        # writes data/data.json
python3 -m http.server 8000          # serve the static site
# open http://localhost:8000
```

No dependencies — pure standard library.

## Notes & caveats

- The seed CSV at `data/raw_layoffs.csv` is **placeholder data, not real layoffs** — it exists only so the chart isn't blank if every remote source fails. The first successful CI run replaces `data.json` with real, aggregated data.
- The model-release timeline is curated by hand — if a date is off, edit `data/markers.json` and commit. Pull requests welcome.
- "Industry" labels come straight from the source CSV; they're not perfectly normalised across years.
- GitHub's free tier allows ~2,000 Actions minutes / month for private repos; this workflow takes ~30s per run, so well within free limits.
